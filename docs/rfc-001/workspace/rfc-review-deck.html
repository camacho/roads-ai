Chunk A · Decision context · §1

Summary — what Trestle is

  • Keeps novice developers unblocked when no mentor is available, without handing over solutions immediately.
  • MVP = pinned open-source coding CLI (pi) + fixed teaching prompt + one adapted skill + a Roads-managed LAN model server + one thin bespoke extension owning attempt state and guardrails.
  • Two restriction strengths: "can't" (no write/execute tools, no configured WAN model path — hard for good-faith use) and "shouldn't" (don't reveal early — model policy, soft).
  • Currently: milestone 0 in progress, next gate is one real pi-to-model turn, no active blockers.

Chunk A · Decision context · §2

Problem — why build this at all

  • Without a mentor, learners wait, use an answer-machine, or struggle unproductively.
  • The cautionary evidence: unfettered GPT-4 helped practice but cut later unassisted exam scores 17% (~1,000 students, E01). Transfer to programming is a hypothesis the pilot tests.
  • Your decision: unblock-first — a solution can be shown after demonstrated effort.
  • Local hosting trades API fees for hardware and upkeep; savings are not assumed.

Chunk A · Decision context · §3

Scope & assumptions — who this is for

  • Learners with minimal professional experience, on small projects, mentor temporarily away.
  • Working assumption to validate: learners can't reliably judge if an explanation is correct.
  • The key distinction: incidental friction (API-name slip → just answer) vs. the actual learning objective (bounds bug → guide). It's relative to the objective, not the bug type.
  • Provisional surface: JS/TS, Node, HTML/CSS, PHP, WebGL. If the Roads leader changes it, a documented procedure regenerates only surface-dependent tests.

Chunk A · Decision context · §4

Goals & non-goals

  • Goals: unblock without solving · reveal progressively · structurally prevent the model writing learner code · run locally on Roads infra · Windows-native low-friction install · honest about uncertainty.
  • Non-goals: preventing determined bypass · registration/auth · grading learners · an IDE extension · topic telemetry in the MVP (without foreclosing it later).

Chunk A · Decision context · §5

Constraints — given vs. chosen

  • Given (yours, non-negotiable): free self-host for <50 engineers · local model, no third-party API · Windows clients on the Roads LAN · no copy-paste solution handover (as qualified by unblock-first) · free at point of use.
  • Chosen (revisable with evidence): graded disclosure over absolute withholding · capability restriction over OS sandboxing · vLLM as the serving engine with the model swappable.
  • Accepted scope: a learner can always open a browser. The design optimizes for being the better option, not preventing bypass.

Chunk B · Product contract · §7

Teaching policy — the product itself

  • Three response modes: DIRECT (incidental friction → answer), LADDER (conceptual → progressively stronger guidance), CLARIFY (vague → one request for context).
  • Full reveal after two substantive attempt events — a deliberately low starting point, labeled hypothesis; the pilot measures over-help, under-help, abandonment.
  • After a reveal: one optional explain-back invitation, no follow-up if ignored.
  • What counts as an "attempt event" is precisely defined with worked transcripts in the companion attempt-evidence doc.

Chunk B · Product contract · §8

User stories — each with its mechanism and proof

  • Learner gets unblocked without an immediately handed-over solution → response modes + reveal gate → exemplar transcripts.
  • Leader gets assurance generated code can't silently enter the repo → no write/execute tools registered, shell escapes rejected → boundary tests.
  • Security admin keeps source on Roads infrastructure → pinned LAN provider, clean credentials → scoped launch checks (literal guarantee needs managed egress).
  • Every story row names the mechanism that delivers it and the evidence that proves it.

Chunk B · Product contract · §9

Data, privacy & the pilot transcript rule

  • No telemetry, no server-side learner data. pi's native sessions persist transcripts locally on the learner's machine only.
  • The extension adds compact decision entries ({mode, attempt_event, attempt_count, new_problem}) — persisted locally in pi's native session files, never in model context (E14), and preserved in opted-in JSONL exports (HTML exports need not display them).
  • Pilot review uses opt-in JSONL exports; the export is the authoritative audit format.
  • Defaults: exports deleted 30 days after pilot end; learners can delete local transcripts anytime.

Chunk C · Delivery · §10

Post-build evaluation — two checks, no vanity metrics

  • Check 1 (bug-cause screen): 20 frozen cases; agents draft, two humans adjudicate labels and freeze the rejection rule before any model output. Passing only rules out obvious failure.
  • Check 2 (Roads validation): authentic learner episodes with what they tried/expected — the real misconception-diagnosis test, run on the production server.
  • No vanity metrics: format-shaped measures (text similarity, prose ratio) are explicitly excluded.
  • If nothing passes: iterate prompt → other model → narrower surface → your scope decision.
Read the RFC text (§10)
Chunk C · Delivery · §11

Pilot protocol — small, supervised, stoppable

  • Small learner cohort, human supervision, pre-agreed stop rules frozen before the first session.
  • Proposed stops: immediate on any privacy/safety event · quality pause at 2 adjudicated-wrong diagnoses in the last 10 reviewed sessions · abandonment pause at 3 of the last 10 ending with no accepted next step and attributed by the learner to Trestle · human check-in for any learner at 3 such abandonments.
  • Privacy before people: transcript custody per §9; consent and export decisions completed before onboarding.

Chunk C · Delivery · §13

Risks — pedagogy ranks above security

  • Teaching failures rank first: because scope is good-faith use, the top risks are wrong diagnosis confidently delivered, over-help that prevents learning, under-help that drives abandonment.
  • Every risk has a detector — the pilot's outcome measures, the unassisted-performance baseline study (design still open — the external −17% result is cautionary evidence, not our threshold), reset-decision review.
  • Watch condition: if learner prior knowledge grows past the assumed level, beginner-calibrated help can invert into harm (expertise reversal, E03).

Cost — allocation, not purchase

Chunk D · Technical design · §15

MVP architecture — locked-down pi + one thin extension

  • The launcher pins everything at start — provider, model, and tools: only read/grep/find/ls registered; write, edit, and shell never exist for the model.
  • The extension owns the teaching machinery: attempt state, pacing, status, shell-escape rejection, and workspace-confined reads.
  • Disclosed residuals: stock /share (needs an authenticated GitHub CLI; creates a secret gist), /login, /model run before extension hooks at this pin (E13) — hence the scoped guarantee: no configured model WAN path at launch, not literal no-internet. Managed egress or a pi patch would make it literal.

Chunk D · Technical design · §16

Key design decisions

  • 16.1 API seam: one OpenAI-compatible endpoint; model artifact = weights + tokenizer + config + template, provisioned by an admin, swappable by configuration.
  • 16.2 Capability bar: what the model must demonstrably do (diagnose causes on the screen) before learner exposure.
  • 16.3 Response choice: one hidden schema-validated pre-check per turn drives mode, attempt counting, and reset; fail-closed to CLARIFY; retained on your decision with a pre-registered removal comparison at milestone 4.
  • 16.4 Isolation: capability restriction, not an OS sandbox — and why that's the right boundary here.
  • 16.5 Uncertainty: honest-uncertainty behavior; no automatic confidence gating (token probabilities aren't correctness probabilities).

Chunk F · RFC 002 · §2

Five principles the selection lives by

  • Derive candidates at execution time — any shortlist written today goes stale (E18); a frozen snapshot records who looked, where, and what was in or out.
  • Measure the teaching task, not leaderboard coding scores.
  • Evaluate the deployed representation — compression (E31), chat template (E32), and serving engine (E33) can each change behavior.
  • Smallest adequate operational choice — quality gates entry; speed and upkeep pick among survivors.
  • Record evidence, not confidence — reproducible manifests, never fluent output.
Read the RFC text (RFC 002 §2)
Chunk F · RFC 002 · §3

What it takes to enter the race

  • License: free for Roads to download, run, and self-host for <50 engineers — read the actual terms, never the label.
  • Runs on the dev Mac; a feasible production representation exists for the Ubuntu/GPU target (the proven run happens at selection).
  • Complete artifact: weights plus tokenizer, config, chat template, helper files. Weights alone are not a candidate.
  • Speaks our API and handles our request sizes — measured, not assumed.

Chunk F · RFC 002 · §4

The selection sequence

  • Screen on the Mac first: the 20-case bug-cause screen against the human-frozen case set; its errors shape the harder Roads validation.
  • Then prove on the real server: deploy the exact production artifact, run parity, run the Roads validation with production authoritative (including the classifier comparison and reset decision), load-test pilot traffic.
  • The screen only rules out failure — usefulness and learning outcomes belong to the supervised pilot.

Chunk F · RFC 002 · §4 · parity

Parity: compare outcomes, not words

  • Mac and server will never match word-for-word — different hardware does floating-point math in different orders (E33).
  • So parity compares what the answer accomplished: right diagnosis, same teaching mode, correct reset decision.
  • Material differences get a human review — and the production result always wins the pilot gate.

Chunk F · RFC 002 · §5–7

Manifest, decision rule, change control

  • Reproducibility manifest per finalist: immutable revisions and hashes for the base model, both artifacts, runtimes, server image digest, request contract, Trestle revisions, surface revision, raw results. Mutable names like latest are never records.
  • Decision rule: survive the pre-registered rules first; then measured tradeoffs — quality, compliance, dev viability, latency and capacity, operational complexity, license risk.
  • Change control: touching the model, quantization, template, image, settings, or teaching prompt reruns the affected gates.
  • The outcome lands in an ADR — the chosen model plus the two architecture decisions (classifier stays? automatic reset ships?).
